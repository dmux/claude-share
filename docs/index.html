How it works

Your files stay yours. Claude runs on the server.

The client uploads a snapshot to an isolated per-session workspace, Claude runs headless there, and every file it changes is diffed and pushed straight back to your local directory.

All traffic flows over a Noise end-to-end encrypted WebSocket.

Why claude-share

Built for a trusted team on a trusted network.

🔐

End-to-end encryption

Noise NNpsk0 (X25519 · ChaCha20-Poly1305 · BLAKE2s): forward secrecy, mutual auth from a shared token, every frame AEAD-sealed.

📁

Directory sync

Content-addressed blob store uploads only what changed; results are diffed per turn and applied atomically to your local files.

💬

Multi-turn REPL

A long-lived session streams assistant output live, just like using Claude Code locally — but powered by the shared instance.

🧩

Hexagonal architecture

A pure core behind ports; WebSocket, the Claude CLI, the filesystem and crypto are swappable adapters. Fully unit-tested.

🛡️

Internal-network only

Refuses to bind a public address without an explicit flag. Path-traversal guards, hash verification, size & session limits.

📦

Cross-platform releases

Prebuilt static binaries for Linux, macOS and Windows (amd64/arm64), shipped automatically by GitHub Actions.

Architecture

Dependencies point inward.

The core knows nothing about WebSocket, os/exec, the filesystem, or Noise — adapters implement the ports.

Security model

Encrypted on the wire. Honest about the endpoints.

  • ✓Forward secrecy — ephemeral X25519 keys per connection.
  • ✓Mutual authentication — a peer without the shared token can't complete the handshake; MITM is rejected.
  • ✓Strong tokens by default — unset CLAUDE_SHARE_TOKEN and the server generates a Bitwarden-style passphrase (~77 bits, crypto/rand) and prints it at startup.
  • ✓Tamper-evident — a single flipped byte fails AEAD decryption and drops the connection.
  • ✓Loopback by default — binding a non-loopback address requires --allow-public.
  • ✓Hardened I/O — path-traversal blocked, blob content hash-verified, file/total/session limits.
Honest scope of “end-to-end.” Claude runs on the server and must read your files, so the server is a legitimate endpoint that handles plaintext. The channel secures the wire between the two real endpoints — it is not a design where the server is blind. If you wouldn't trust the operator with your source, don't use it.
Quickstart

Two processes, one shared secret.

The two ends share one secret, CLAUDE_SHARE_TOKEN. Leave it unset on the server and it prints a generated passphrase to copy to each client — or set your own on both ends. The server machine needs claude installed and logged in.

1

Server — on the machine with Claude

export CLAUDE_SHARE_TOKEN='a-strong-shared-secret'
# bind to a LAN address (internal network only)
claude-share-server --addr 192.168.1.50:8443 --allow-public
2

Client — from any project directory

export CLAUDE_SHARE_TOKEN='a-strong-shared-secret'
claude-share-client --server ws://192.168.1.50:8443/ws --dir .
3

Or build from source

git clone https://github.com/dmux/claude-share
cd claude-share
make build   # -> bin/claude-share-server, bin/claude-share-client
claude-share
